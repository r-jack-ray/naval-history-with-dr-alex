# Bruships Selectable Theme Plan

Timestamp: 2026-07-12T14:16:20-05:00
Status: Planning only; not implemented

## Objective

Add a new selectable site theme named **Bruships**, based on the supplied orange, navy, blue, white, and soft-bokeh visual treatment.

Bruships will be a fourth explicit choice alongside **Light**, **Dark**, and **System**. It will not replace or change any existing theme or the current default behavior.

## Confirmed Constraints

- Preserve Light, Dark, and System as existing selectable choices.
- Keep System resolving only to Light or Dark from the operating-system preference.
- Continue saving the visitor's explicit choice under the existing `naval-history-theme` local-storage key.
- Ignore the YouTube `Live in 20 days / Notify me` overlay in the reference image.
- Do not ship the supplied raster screenshot or its embedded wording as a site background.
- Keep the initial treatment static and CSS-based; no animated bokeh or parallax.
- Do not change transcripts, curated segment shards, generated archive data, Pagefind data, or other content-pipeline outputs.
- Avoid full site builds while implementing the change whenever focused checks and browser verification provide sufficient evidence. Reserve a full build for a final one-time gate or a problem that cannot be validated more narrowly.

## Visual Direction

| Role | Starting color | Intended use |
| --- | --- | --- |
| Orange | `#FA7E26` | Masthead and primary Bruships accent |
| Bright orange | `#FF852F` | Small highlights and gradient variation |
| Peach | `#FFB27A` | Soft glow and secondary highlights |
| Deep navy | `#08347D` | Main page canvas |
| Dark blue | `#0A4194` | Cards and content surfaces |
| Vivid blue | `#0A5BAD` | Bokeh, hover, and selected-state accents |
| White | `#FEFCFD` | Primary text on navy and blue |
| Dark ink | `#0B1F3A` | Accessible text on orange |

Translate the reference's orange-over-navy split into an orange sticky masthead over a navy study-guide canvas. Recreate the soft circles with low-opacity CSS radial gradients, confined to decorative chrome and open background areas. Keep reading surfaces opaque or strongly scrimmed so bokeh never competes with article, search, Q&A, or evidence text.

Use rounded, broadcast-like styling with approximately 12-16px card radii, pill-like compact controls, strong headings, and restrained tracking on short labels. Retain the existing readable body type and normal casing for long video and segment titles.

## Current Implementation Surface

- `site/src/layouts/BaseLayout.astro` owns the shared header, theme picker, early pre-paint initialization, runtime theme state, local-storage persistence, and System preference listener.
- `site/src/styles/site.css` owns the Light and Dark token sets, theme-switcher layout and icons, shared component styling, and responsive header behavior.
- Every public Astro route uses `BaseLayout`, so the theme can be applied site-wide without page-specific theme state.
- `README.md` currently documents only Light, Dark, and System.
- There are no dedicated theme persistence, DOM, visual-regression, or contrast tests.

## Implementation Plan

### 1. Extend the theme state without changing existing choices

Update `site/src/layouts/BaseLayout.astro`:

- Add `bruships` to both valid-choice sets used by the early head script and the runtime script.
- Add a fourth accessible picker button labelled `Bruships` with `data-theme-choice="bruships"`, an explicit `aria-label`, and the existing `aria-pressed` behavior.
- Add a small orange/navy split icon for the new choice.
- Preserve `data-theme-choice="bruships"` and apply `data-theme="bruships"` when selected.
- Keep unknown stored values falling back to System.
- Keep the operating-system change listener active only while System is selected.
- Preserve the early initialization path so a saved Bruships choice is applied before the page body renders.

Do not assign the custom string `bruships` directly to the browser's native `color-scheme` property. Resolve native form controls to `dark` for Bruships while retaining the custom data attribute for CSS selection.

### 2. Add semantic tokens and the Bruships palette

Update `site/src/styles/site.css`:

- Add `:root[data-theme="bruships"]` beside the existing Light and Dark token definitions.
- Define every existing color token for Bruships: ink, muted text, borders, paper, panels, header, signals, accent, image placeholders, controls, active controls, and feature shadow.
- Add header-specific ink, muted, hover, border, and control tokens so the orange header can use dark navy text while the blue page uses white text.
- Add semantic radius tokens for cards and controls. Keep their current values for Light and Dark, then give Bruships the rounder values.
- Add theme-scoped decorative background tokens or selectors for orange and blue bokeh layers.

This token refactor must leave the rendered Light and Dark palettes and geometry unchanged.

### 3. Apply the Bruships visual treatment

- Give the sticky header an orange base with restrained peach radial glows.
- Give the page canvas a deep navy base with sparse blue radial glows.
- Use solid or strongly opaque dark-blue panels for cards, filters, search results, source passages, and evidence text.
- Use orange for selected accents and small emphasis, not for long text.
- Use white on navy/blue and dark navy on orange.
- Keep bokeh static, decorative, non-interactive, and outside the accessibility tree.
- Avoid fixed or parallax backgrounds, particularly on mobile.

Do not include a ship silhouette in the first implementation. If one is later requested, create a small repository-owned SVG rather than extracting it from the reference screenshot.

### 4. Adapt the picker for four choices

- Change the desktop and mobile switcher grids from three to four choices.
- Keep visible labels on wide screens.
- Add an intermediate responsive rule that collapses labels before the four-button control can collide with the brand or navigation.
- Preserve the existing top-right mobile placement and horizontal navigation row.
- Keep accessible names available when visible labels are hidden.
- Avoid horizontal page scrolling and keep targets comfortably usable.

Explicitly inspect widths around 320, 360, 375, 760, 761, 768, 980, and a wide desktop viewport.

### 5. Strengthen accessibility for the new theme

- Do not use white for normal text on the sampled orange; it does not provide sufficient contrast.
- Use `#0B1F3A` or an equally dark navy for header text on orange.
- Use white or a verified pale tint for primary text on deep navy and blue panels.
- Verify muted text, borders, links, focus rings, selected controls, and disabled states independently.
- Ensure selected theme state and interactive focus do not depend only on color or background imagery.
- Keep bokeh unanimated. If motion is introduced in a later change, disable it under `prefers-reduced-motion`.
- Provide a usable forced-colors fallback that removes decorative gradients without hiding selected or focused states.

### 6. Document and validate

- Update `README.md` to list Bruships alongside Light, Dark, and System.
- Do not edit generated site archive files.
- During implementation, use the focused Astro check and browser verification instead of repeatedly running `site:build`, `site:build:generated`, or `site:build:full`.
- Run a full site build only when a focused check cannot cover a material risk or as a single final verification gate when required.
- Use the fixed Windows npm executable for the focused validation path:

```powershell
C:\Program Files\nodejs\npm.cmd run site:check:generated
git diff --check
```

## Manual Verification Matrix

Check Home, Videos, individual video guides, Time Notes, individual time notes, Topics, individual topic pages, and Search in all four choices.

For Bruships, verify:

- Selection applies immediately and sets the correct pressed state.
- The choice survives reloads, navigation, and deep-route entry.
- A saved Bruships choice does not flash Light or Dark during first paint.
- Operating-system light/dark changes do not override an explicit Bruships choice.
- Returning to System resumes operating-system preference tracking.
- Forms, search controls, cards, embedded media, source panels, pagination, and empty states remain readable.
- Keyboard focus remains visible against orange, navy, blue, and white surfaces.
- The sticky header remains legible over page content.
- No tested viewport develops clipped controls or horizontal scrolling.
- Light, Dark, and System remain visually and behaviorally unchanged.

## Acceptance Criteria

- The picker offers Light, Dark, Bruships, and System.
- Bruships is persisted as an explicit choice without altering the existing default.
- System continues to resolve only to Light or Dark.
- The theme expresses the reference through orange/navy division, rounded blue surfaces, and static bokeh.
- Text and controls meet appropriate contrast and focus requirements.
- The YouTube scheduling overlay and reference raster are absent.
- All major public routes work at desktop, tablet, and mobile widths.
- Focused site checks and `git diff --check` pass. If a final full site build is necessary, it is run once and passes rather than being repeated during implementation.
